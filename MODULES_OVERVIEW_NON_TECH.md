# Короткий опис модулів проєкту

Цей документ коротко пояснює, які основні частини є в backend starter-проєкті та для чого вони потрібні. Опис написаний простою мовою для нетехнічних спеціалістів: менеджерів, замовників, аналітиків або членів команди, яким потрібно розуміти загальну структуру системи.

---

## 1. API-додаток

**Для чого потрібен:**  
API — це основна точка входу для сайту, мобільного додатку або іншого клієнта.

Через API користувачі можуть створювати акаунти, отримувати дані, надсилати форми, виконувати дії в системі та взаємодіяти з backend.

**Простими словами:** це “двері”, через які frontend спілкується з сервером.

---

## 2. Worker-додаток

**Для чого потрібен:**  
Worker виконує фонові задачі, які не потрібно робити прямо під час запиту користувача.

Наприклад:

- відправка email;
- обробка черг;
- повторна обробка задач після помилки;
- виконання довгих операцій у фоні.

**Простими словами:** це “помічник”, який робить важку або довгу роботу окремо, щоб API працював швидко.

---

## 3. Cron-додаток

**Для чого потрібен:**  
Cron-додаток запускає задачі за розкладом.

У поточній реалізації він раз на хвилину ставить у чергу команду на обробку outbox-подій.

Cron сам не читає таблицю outbox і не виконує фонову бізнес-логіку.

Фактичну обробку виконує Worker-додаток.

Наприклад:

- перевірити незавершені події;
- очистити старі записи;
- запустити регулярну синхронізацію;
- перевірити стан системи.

**Простими словами:** це “планувальник”, який запускає потрібні дії автоматично у визначений час.

---

## 4. Domain-модуль

**Для чого потрібен:**  
Domain містить основні бізнес-правила проєкту.

Тут описується, що таке користувач, замовлення, баланс, статус, подія тощо.

**Простими словами:** це “серце бізнес-логіки”, де зберігаються головні правила системи.

---

## 5. Application-модуль

**Для чого потрібен:**  
Application описує конкретні сценарії роботи системи.

Наприклад:

- створити користувача;
- отримати користувача;
- поповнити баланс;
- створити замовлення;
- виконати певну бізнес-операцію.

**Простими словами:** це “інструкції”, які пояснюють системі, як виконувати конкретні дії.

---

## 6. Contracts-модуль

**Для чого потрібен:**  
Contracts описує домовленості між частинами системи.

Наприклад, application знає, що є “сховище користувачів”, але не знає, як саме воно працює всередині.

**Простими словами:** це “угоди” між модулями, щоб частини системи не залежали одна від одної напряму.

---

## 7. Infrastructure-модуль

**Для чого потрібен:**  
Infrastructure містить реальні підключення до зовнішніх сервісів і технічних інструментів.

Наприклад:

- база даних;
- Redis;
- черги;
- email;
- файлове сховище;
- кеш;
- логування.

**Простими словами:** це “технічний шар”, який підключає систему до реального світу.

---

## 8. Shared-модуль

**Для чого потрібен:**  
Shared містить спільні допоміжні речі, які можуть використовуватись у різних частинах проєкту.

Наприклад:

- спільні типи;
- утиліти;
- формат пагінації;
- базові DTO;
- загальні константи.

**Простими словами:** це “спільна коробка з інструментами” для всього проєкту.

---

# Інфраструктурні модулі

---

## 9. Config Module

**Для чого потрібен:**  
Відповідає за налаштування проєкту.

Наприклад:

- порт додатку;
- доступ до бази даних;
- налаштування Redis;
- налаштування email;
- секрети для авторизації.

**Простими словами:** це “центр налаштувань”, щоб усі важливі параметри були в одному місці.

---

## 10. Logger Module

**Для чого потрібен:**  
Відповідає за логи системи.

Логи допомагають бачити:

- що відбулося;
- коли відбулося;
- де виникла помилка;
- який запит або процес її викликав.

**Простими словами:** це “журнал подій”, який допомагає розробникам швидше знаходити проблеми.

Для кожного HTTP-запиту автоматично створюється `requestId`.

Якщо один бізнес-процес проходить через кілька сервісів, для нього може використовуватися спільний `correlationId`.

Ці значення автоматично додаються до логів і повертаються клієнту в response headers.

---

## 11. Database / PostgreSQL + Drizzle Module

**Для чого потрібен:**  
Відповідає за роботу з базою даних.

У базі зберігаються користувачі, події, журнали, ключі ідемпотентності, outbox-події та інші дані.

**Простими словами:** це “памʼять системи”, де зберігаються важливі дані.

---

## 12. Redis Module

**Для чого потрібен:**  
Redis використовується для швидкого зберігання тимчасових даних.

Наприклад:

- кеш;
- ліміти запитів;
- блокування процесів;
- технічні дані для черг.

**Простими словами:** це “дуже швидке тимчасове сховище”.

---

## 13. BullMQ / Queue Module

**Для чого потрібен:**  
Черги дозволяють ставити задачі в обробку і виконувати їх у фоні.

Наприклад:

- відправити email;
- обробити подію;
- виконати задачу з повторними спробами;
- перенести важку роботу з API у worker.

**Простими словами:** це “черга задач”, щоб система не намагалась зробити все одразу.

---

## 14. Cache Module

**Для чого потрібен:**  
Кеш зберігає часто використовувані дані, щоб не запитувати їх постійно з бази.

Це допомагає:

- пришвидшити роботу API;
- зменшити навантаження на базу;
- швидше віддавати дані користувачу.

**Простими словами:** це “швидка памʼять” для популярних даних.

---

## 15. Health Module

**Для чого потрібен:**  
Health module показує, чи працює система і її основні залежності.

Health Module має окрему просту перевірку того, що процес запущений, та окрему перевірку готовності PostgreSQL, Redis і черг.

Якщо залежності недоступні, readiness endpoint повертає помилку, щоб Docker або Kubernetes не направляли трафік на несправний instance.

Перевіряє:

- сам додаток;
- базу даних;
- Redis;
- черги.

**Простими словами:** це “індикатор здоровʼя системи”.

---

## 16. Exception Module

**Для чого потрібен:**  
Відповідає за єдиний формат помилок у системі.

Замість різних випадкових повідомлень API повертає зрозумілий і однаковий формат помилки.

**Простими словами:** це “єдиний стандарт помилок”, щоб frontend і команда легше їх обробляли.

---

## 17. Mail Module

**Для чого потрібен:**  
Відповідає за відправку email.

Наприклад:

- welcome email;
- підтвердження реєстрації;
- відновлення паролю;
- системні повідомлення;
- повідомлення про важливі дії.

**Простими словами:** це “модуль для листів”.

---

## 18. Storage Module

**Для чого потрібен:**  
Відповідає за зберігання файлів.

Наприклад:

- аватари;
- документи;
- зображення;
- вкладення;
- експортовані файли.

Файли можуть зберігатися локально або в S3/Minio-сумісному сховищі.

**Простими словами:** це “файлове сховище системи”.

---

## 19. Transaction Module

**Для чого потрібен:**  
Транзакції гарантують, що кілька важливих дій або виконаються всі разом, або не виконається жодна.

Наприклад, якщо потрібно:

- створити користувача;
- записати аудит;
- змінити баланс;
- створити повʼязаний запис.

Якщо одна дія впала, система може відкотити всі попередні зміни.

**Простими словами:** це “захист від напівзбережених операцій”.

---

## 20. Domain Events

Окремого Event Bus Module у поточній версії немає.

Бізнес-події зберігаються через Outbox Module у базі даних.

---

## 21. Outbox Module

**Для чого потрібен:**  
Outbox Module забезпечує надійну доставку важливих domain та integration events.

Подія записується в таблицю outbox_events у тій самій PostgreSQL-транзакції, що й основна бізнес-зміна. Завдяки цьому система не потрапляє в ситуацію, коли бізнес-дані вже збережені, але повідомлення для worker не було створене через тимчасову недоступність Redis або BullMQ.

Порядок роботи:

Application use case відкриває транзакцію.

- У межах транзакції зберігаються бізнес-дані.
- У межах тієї самої транзакції записується outbox event.
- Cron або worker вибирає pending events невеликими пакетами.
- Записи блокуються через PostgreSQL FOR UPDATE SKIP LOCKED.
- Кожна подія додається у відповідну BullMQ queue.
- Після успішної публікації outbox event позначається як processed.
- При помилці збільшується кількість attempts і встановлюється час наступної спроби.
- Після перевищення максимальної кількості спроб event переходить у failed.

BullMQ job використовує стабільний jobId, побудований на основі outbox event id. Це зменшує ризик дублювання job при повторній обробці.

Outbox гарантує надійну доставку за моделлю at-least-once. Тому event handlers також повинні бути idempotent.

**Простими словами:** це “страховка для важливих подій”.

---

## 22. Audit Module

**Для чого потрібен:**  
Audit записує важливі дії користувачів або системи.

Наприклад:

- хто створив запис;
- хто змінив дані;
- хто виконав важливу дію;
- коли це сталося.

**Простими словами:** це “історія важливих дій у системі”.

---

## 23. Rate Limiter Module

**Для чого потрібен:**  
Rate limiter обмежує кількість запитів за певний час.

Це захищає систему від:

- спаму;
- brute-force атак;
- надмірного навантаження;
- зловживань API.

**Простими словами:** це “обмежувач частоти запитів”.

---

## 24. Distributed Lock Module

**Для чого потрібен:**  
Distributed lock не дозволяє кільком процесам одночасно виконувати одну і ту саму критичну задачу.

Особливо корисно, коли система запущена в кількох копіях.

Наприклад:

- не запускати один cron двічі;
- не обробляти одну задачу паралельно;
- уникати дублювання важливих операцій.

**Простими словами:** це “замок”, який дозволяє виконувати задачу тільки одному процесу.

---

## 25. Idempotency Module

**Для чого потрібен:**  
Idempotency захищає від повторного виконання однієї і тієї ж дії.

Наприклад:

- користувач натиснув кнопку двічі;
- frontend повторив запит;
- платіжний сервіс надіслав webhook повторно;
- worker повторив задачу.

Система розуміє, що така дія вже була, і не створює дубль.

**Простими словами:** це “захист від випадкових повторів”.

# Приклад бізнес-модуля

---

## 26. Users Module

**Для чого потрібен:**  
Цей модуль надає базові можливості для роботи з користувачами та їх авторизацією.

На цей момент у starter-проєкті реалізовано:

- реєстрацію нового користувача;
- вхід за email і паролем;
- отримання даних поточного авторизованого користувача;
- хешування паролів;
- перевірку авторизації;
- перевірку ролей користувача;
- підтримку JWT-авторизації;
- підтримку серверних сесій через Redis.

Режим авторизації можна вибрати через налаштування проєкту.

У JWT-режимі клієнт отримує короткоживучий access token і довгоживучий refresh token.

Після кожного оновлення авторизації система видає нову пару токенів, а попередній refresh token стає недійсним.

Якщо старий refresh token буде використаний повторно, система відкликає всю пов’язану авторизаційну сесію та вимагає повторного входу.

Під час виходу refresh token відкликається, а access token блокується до завершення строку його дії.

У session-режимі backend створює сесію в Redis і передає клієнту захищену cookie.

Starter містить базову модель користувача та repository для PostgreSQL. Додаткові бізнес-можливості, наприклад профіль користувача, баланс, налаштування, блокування або підтвердження email, повинні додаватися окремо відповідно до вимог конкретного проєкту.

Важливі події, пов’язані зі створенням користувача, можуть передаватися через Outbox і оброблятися у worker. Наприклад, після реєстрації worker може надіслати welcome email.

# Підсумок

У цьому starter-проєкті вже є базова backend-платформа для майбутніх продуктів.

Вона включає:

- окремий API для запитів;
- окремий Worker для фонових задач;
- окремий Cron для задач за розкладом;
- базу даних;
- Redis;
- черги;
- кеш;
- email;
- файлове сховище;
- логування;
- перевірку здоровʼя системи;
- захист від помилок, повторів і дублювання;
- аудит дій;
- основу для масштабування.

**Головна користь:** цей starter дозволяє не створювати базову інфраструктуру з нуля для кожного нового проєкту, а одразу починати розробку бізнес-функціоналу.
